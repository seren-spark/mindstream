# 混合检索模块设计

> 对齐：`向量存储模块设计.md`、`向量化模块设计.md`、`文本切片模块设计.md`  
> 当前实现：`backend/app/services/retrieval_service.py`

---

## 1. 模块职责定义

### 在 RAG 链路中的位置

```
用户 query → 【混合检索】 → Top-K context → Prompt 构造 → LLM 回答 + 引用
                ↑ 本模块
```

检索模块负责：从知识库中找出与 query **最相关** 的 chunk，作为 LLM 的上下文与引用来源。

### 为什么检索质量决定回答上限

> **模型只能基于检索到的 context 回答。** 检索漏召 → 模型缺信息会胡编；检索噪声多 → 回答跑题。RAG 的天花板由「召回 + 精度」共同决定，而不是模型参数 alone。

### 为什么不能只依赖单一检索

| 方式 | 优势 | 劣势 |
|------|------|------|
| **仅向量** | 语义相似、改写 query 也能命中 | 专有名词、编号、错误码精确匹配弱 |
| **仅关键词** | 术语、型号、政策编号精确 | 同义改写、口语化 query 漏召 |
| **混合 + RRF** | 两路互补、实现轻量 | 需融合逻辑（Demo 可接受） |

## 2. 检索流程设计

```
query
  ├─→ 向量召回 (Chroma, top recall_k) ──→ ranked list A
  ├─→ 关键词召回 (SQLite chunks, top recall_k) ──→ ranked list B
  ├─→ 按 chunk_id 去重
  ├─→ RRF 融合 → 统一 score
  ├─→ 回表 enrichment（source_location、display_title）
  └─→ 裁剪 top_k → RetrievalHit[]
```

| 阶段 | 说明 |
|------|------|
| 召回 | 每路取 `recall_k`（默认 20），留足融合空间 |
| 去重 | 以 `chunk_id` 为键 |
| 融合 | RRF，k=60 |
| 裁剪 | 最终 `top_k`（默认 5）给 RAG prompt |
|  enrichment | 从 SQLite chunks + items 补全引用字段 |

## 3. 关键词检索设计

### Demo 方案：SQLite `ILIKE`

```sql
SELECT chunks.* FROM chunks
JOIN knowledge_items ON ...
WHERE knowledge_base_id = ? AND item.status = 'ready'
  AND chunks.content ILIKE '%关键词%'
LIMIT recall_k
```

### 多词策略

1. 优先整句 `LIKE %query%`
2. 无结果时按空格/标点切词，**任一词命中**即召回（OR）
3. 排序：命中词数多者优先

### 全文索引（FTS5）

| 方案 | Demo | 生产 |
|------|------|------|
| LIKE | ✅ 零迁移、够用 | 大数据慢 |
| SQLite FTS5 | 后续 | 中文需 jieba 分词表 |
| Elasticsearch | ❌ 过重 | 大规模 |

## 4. 向量检索设计

- **Query embedding：** `EmbeddingService.embed_query(query)`（共享 cache）
- **Chroma 查询：** `VectorStoreService.query(kb_id, query, filter, top_k=recall_k)`
- **Metadata 过滤：** `item_status=ready`、category、tag（tag 先查 SQLite item_ids）
- **分数：** `1 - cosine_distance`

---

## 5. 融合与重排设计

### RRF 原理

```
RRF_score(d) = Σ  1 / (k + rank_i(d))
```

- `rank_i(d)`：文档 d 在第 i 路召回中的排名（从 1 开始）
- `k`：常数，默认 60，平滑头部排名差异
- **无需**统一向量分与 BM25 分量纲

### 为什么适合 Demo

- 实现 < 30 行，无额外模型
- 不依赖训练数据
- 两路召回互补效果明显

### 复杂 Reranker（Demo 不做）

| Cross-encoder / Cohere Rerank | 问题 |
|-------------------------------|------|
| 精度更高 | 多一次 API、延迟与成本 |
| | Demo 先用 RRF 验证链路 |

## 6. 返回结果结构 `RetrievalHit`

| 字段 | 用途 |
|------|------|
| `chunk_id` | 引用脚注关联 |
| `content` | RAG prompt context |
| `score` | RRF 融合分 |
| `source_name` | 文档标题 |
| `display_title` | 卡片标题（含章节） |
| `source_location` | 页码/偏移/heading |
| `knowledge_item_id` | 跳转条目 |
| `highlight_text` | 前端摘要高亮 |
| `recall_sources` | `vector` / `keyword`（调试） |

## 7. API

```
POST /api/knowledge-bases/{id}/search
```

请求体与 `HybridSearchRequest` 一致；原 `vectors/search` 保留为纯向量调试入口。

## 8. 面试表达

> 我们做了关键词 + 向量双路召回，用 RRF 融合，避免单一检索的盲区：向量负责语义，关键词负责工号、条款编号等精确命中。召回 20、融合后 top 5 给 LLM，结果带 chunk_id 和 source_location，直接进 prompt 和引用卡片。Demo 没上 cross-encoder rerank，先把完整检索链路跑通。

## 9. 验证清单

1. 精确术语 query 关键词路有命中
2. 口语化 query 向量路有命中
3. 混合结果 score 为 RRF 分，无重复 chunk_id
4. 返回含 source_location、highlight_text
5. disabled 条目不被召回
