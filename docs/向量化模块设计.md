# 向量化模块设计

> 对齐：`文本切片模块设计.md`、`向量存储模块设计.md`  
> 当前实现：`embedding_service.py` + `indexing_service.py`

---

## 1. 模块职责定义

### 在 RAG 链路中的位置

```
解析 → 切片(Chunk) → 【向量化(Embedding)】 → 向量库(ChromaDB) → 检索 → 回答
                         ↑ 本模块
```

向量化模块负责：把 `ChunkDraft.content` 批量转为 embedding 向量，并交给向量库存储层写入。

### 为什么不只是一次 API 调用

| 问题 | 说明 |
|------|------|
| **成本** | 按 token/请求计费，逐条调用费用高 |
| **限流** | 云 API 有 QPS/RPM 限制，需 batch + 重试 |
| **可靠性** | 网络抖动、429 需退避重试，不能拖垮整条 pipeline |
| **重复计算** | 相同文本 re-process 应走缓存，避免重复 embed |
| **状态联动** | 失败需写 `failed` + `error_message`，成功才 `ready` |
| **职责分离** | embed 不管 ChromaDB；向量库不管 batch/retry |

## 2. 输入输出设计

### 输入 `EmbeddingInput`

| 字段 | 说明 |
|------|------|
| `chunk_id` | 与 SQLite / Chroma id 对齐 |
| `text` | chunk.content（含标题前缀） |
| `text_hash` | SHA256(text)，缓存键 |
| `knowledge_item_id` | 条目关联 |
| `order_index` | 批次内排序 |

### 输出 `EmbeddingRecord`

| 字段 | 说明 |
|------|------|
| `chunk_id` | 对应输入 |
| `text_hash` | 缓存键 |
| `vector` | float[] |
| `dimension` | 向量维度 |
| `from_cache` | 是否命中缓存 |

### 批次结果 `EmbeddingBatchResult`

| 字段 | 说明 |
|------|------|
| `records` | 全部向量记录 |
| `total` | 处理条数 |
| `cache_hits` | 缓存命中数 |
| `api_calls` | 实际 API 调用批次数 |
| `model_name` | 当前 embedding 模型标识 |

### 与 chunk / 条目绑定

```
KnowledgeItem (1) → ChunkDraft (N) → EmbeddingInput (N) → EmbeddingRecord (N)
                                              ↓
                              IndexingService → VectorStoreService.upsert_vectors
```

## 3. 批量处理方案

### 推荐参数（Demo）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `embedding_batch_size` | **32** | 每批 chunk 数，OpenAI 常见上限 2048 inputs |
| `embedding_max_retries` | **3** | 单批最大重试 |
| `embedding_retry_base_delay` | **1.0s** | 指数退避基数 |

### 切 batch 策略

```python
for batch in chunks(inputs, batch_size=32):
    vectors = embed_with_retry(batch)  # 一批一次 API
    results.extend(vectors)
```

- 按 `order_index` 排序后顺序切 batch，保证确定性
- 单条目 chunk 数 < batch_size 时，一次请求完成
- 多条目 rebuild 时按条目顺序 batch，不跨条目乱序（便于进度追踪）

### 并发控制（Demo）

- **同步串行 batch**：Demo 够用，逻辑简单
- 后续可加 `asyncio.Semaphore` 限制并发批次数

### 失败重试

```
attempt 0 → 失败 → sleep 1s
attempt 1 → 失败 → sleep 2s
attempt 2 → 失败 → sleep 4s
attempt 3 → 抛 EmbeddingPipelineError
```

- 429 / 5xx / 超时：重试
- 4xx（除 429）：不重试，直接失败

## 4. 缓存设计

### 是否需要

**需要。** 同一条目 re-process、rebuild 索引、内容未变的 chunk 不应重复 embed。

### 缓存键

```
{model_name}:{text_hash}
```

`model_name` 随 embedding 后端变化（`hash` / `chromadb_default` / `text-embedding-3-small`）。

### Demo 实现

- 本地 JSON 文件：`./data/embedding_cache.json`
- 内存 dict + 写时落盘
- **不做** Redis / 独立 DB 表（后续可换）

### 收益 vs 复杂度

| 收益 | 复杂度 |
|------|--------|
| 节省 API 费用 | 低（JSON 文件） |
| 加快 re-process | 需 model 变更时清缓存 |
| rebuild 幂等 | 缓存文件需 gitignore |

## 5. 状态流设计

```
pending → processing → ready
              ↓
            failed → pending（重试 process）
```

| 阶段 | processing_progress | 说明 |
|------|---------------------|------|
| 开始处理 | 10 | 解析/校验 |
| 解析完成 | 30 | 有 raw_text |
| 切片完成 | 60 | chunks 落 SQLite |
| 向量化中 | 80 | batch embed |
| 完成 | 100 | 向量写入 Chroma，`ready` |
| 失败 | 0 | `failed` + `error_message` |

### 失败处理

- 捕获 `EmbeddingPipelineError`
- 写 `item.error_message = "向量化失败: ..."`
- 删除该条目已写入的部分向量（幂等清理）
- **保留** SQLite chunks（便于排查，re-process 可复用）

### 重试

`POST /knowledge-items/{id}/process`：`failed → processing → ...`

## 6. 与向量库的接口边界

| 模块 | 负责 | 不负责 |
|------|------|--------|
| **EmbeddingService** | batch embed、retry、cache、model 切换 | ChromaDB、条目状态 |
| **IndexingService** | 编排 chunk→embed→upsert、进度更新 | 切片算法 |
| **VectorStoreService** | upsert/delete/query collection | 调用 embedding API |

```
IndexingService.index_item(item, drafts):
    records = EmbeddingService.embed_chunks(inputs)   # 向量化
    VectorStoreService.upsert_vectors(item, drafts, vectors)  # 存储
```

## 7. 代码结构

```
backend/app/
  schemas/embedding.py
  services/embedding_service.py    # batch + retry + cache
  services/embedding_cache.py      # JSON 缓存
  services/indexing_service.py     # pipeline 编排
```

### API（沿用现有）

- 条目 process 自动触发向量化
- `POST .../vectors/rebuild` 批量 re-index（内部走 IndexingService）

---

## 8. 面试表达

> 向量化不是一次 API 调用，而是带 batch、retry、cache 的 pipeline。我们按 32 条一批 embed，失败指数退避重试 3 次，text hash + model 做本地缓存避免重复计费。EmbeddingService 只管产出向量，VectorStoreService 只管写入 ChromaDB，IndexingService 编排两者并与条目状态机联动——embed 失败条目进 failed，chunks 保留便于重试。

## 9. 验证清单

1. 条目 process 后 Chroma 向量数 = SQLite chunk 数
2. 相同内容 re-process，日志显示 cache_hits > 0
3. 模拟 embed 失败，条目 status=failed 且有 error_message
4. failed 条目 re-process 可恢复 ready
5. rebuild 不重复调用 API（缓存命中）
