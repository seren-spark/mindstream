# 文本切片模块设计

> 对齐：`document-parser-module.md`、`knowledge-item-module.md`  
> 当前实现：`backend/app/services/chunk_service.py`（section + 段落 + 滑窗混合策略）

---

## 1. 模块职责定义

### Chunk 在 RAG 链路中的位置

```
文件 → 解析(ParseResult) → 【切片(Chunk)】 → 向量化 → 检索 → LLM 回答 + 引用
                              ↑ 本模块
```

解析输出的是**文档级全文**（`knowledge_item.content`），而 embedding 与向量检索工作在**固定长度、语义相对独立的片段**上。Chunk 是把「整篇文档」变成「可检索单元」的转换层。

| 阶段 | Chunk 的作用 |
|------|-------------|
| **解析** | 消费 `raw_text` + `sections`，不碰原始文件 |
| **Embedding** | 每条 chunk 独立编码，长度控制在模型上下文窗口内 |
| **检索** | 向量库最小检索粒度，query 与 chunk 做相似度匹配 |
| **回答** | Top-K chunk 作为 LLM context，每条 chunk 对应一条 reference 卡片 |

**没有合理切片时的问题：**

- 整篇 embedding → 向量语义被稀释，细粒度问题召回差
- 整篇塞给 LLM → 超出 context、噪声多、引用无法精确定位

## 2. 切片策略设计

### 推荐参数（中文知识库 Demo）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `chunk_size` | **400 字符** | 中文约 200–350 字 ≈ 1 段完整语义 |
| `overlap` | **60 字符**（≈15%） | 避免句首/句尾截断后丢失上下文 |
| `min_chunk_size` | **80 字符** | 过短片段合并到相邻块 |

配置项见 `backend/app/core/config.py`：`chunk_size`、`chunk_overlap`、`chunk_min_size`。

### 为什么不能切太大 / 太小

| 切太大 | 切太小 |
|--------|--------|
| 向量语义稀释，细粒度 query 召回低 | 指代丢失（「它」「上述」无上下文） |
| Top-K 常召回整章，LLM 噪声多 | 同一知识点碎片化，Top-K 重复 |
| 引用卡片展示 800 字，用户难定位 | 卡片只有半句话，不可读 |
| 5 条 chunk 占满 context，信息密度低 | chunk 数量膨胀，存储与检索成本上升 |

### 策略：混合切分

```
优先级：sections 边界 > 段落边界 > 句子边界 > 字符滑窗
计量单位：Demo 用字符；生产环境换 tiktoken 按 token 计量
```

| 维度 | 选择 | 理由 |
|------|------|------|
| **计量** | 字符（Demo）/ Token（生产） | 与 embedding 模型窗口一致 |
| **边界** | section → 段落 → 句子 → 滑窗 | 纯滑窗快但易断语义；混合兼顾质量与成本 |
| **中文** | `\n\n` 段落、`。！？；` 句子 | 中文无空格分词，句末标点更可靠 |

**中文语料注意：**

1. 不在固定字数 mid-字符 硬切，句子边界回退
2. 标题注入 chunk 前缀：`【安装指南 > 环境要求】\n正文...`
3. 全角/半角标点 normalize
4. 列表项「序号 + 内容」尽量同块
5. 表格按行组切，保留表头

## 3. 输出结构设计

### Pydantic Schema（`schemas/chunk.py`）

```python
SourceLocation   # page、section、char 偏移、heading 面包屑
ChunkMetadata    # 策略版本、chunk_size、is_continued 等
ChunkDraft       # 切片输出，写入 DB + 送 embedding
ChunkResponse    # API 响应
```

### 字段说明

| 字段 | 用途 |
|------|------|
| `chunk_id` | UUID，DB 主键 / ChromaDB document id / RAG reference 关联键 |
| `knowledge_item_id` | 回溯源文档条目 |
| `content` | embedding 输入 + 引用卡片摘要（含标题前缀） |
| `order_index` | 文档内顺序，支持上下文扩展 |
| `token_count` | 监控、限流、成本估算 |
| `source_location` | 引用卡片核心：页码、章节、字符偏移 |
| `metadata` | 策略版本、是否续块、文件类型 |

### DB 表 `chunks`

| 列 | 类型 | 说明 |
|----|------|------|
| `id` | VARCHAR(36) PK | chunk_id |
| `knowledge_item_id` | FK | 级联删除 |
| `knowledge_base_id` | INT | 冗余，便于按库查询 |
| `content` | TEXT | 含标题前缀的正文 |
| `order_index` | INT | 0, 1, 2... |
| `token_count` | INT | 估算 token 数 |
| `source_location` | JSON | 定位信息 |
| `chunk_metadata` | JSON | 策略元数据 |

## 4. 切片规则细化

### 超长段落

1. 按句子切分（`。！？；`）
2. 贪心合并直到接近 `chunk_size`
3. 仍超长 → 字符滑窗 + `overlap`
4. 标记 `metadata.is_continued = True`

### 保留标题层级

- `heading_path` 来自 `ParseSection.title` + 标题栈
- `content` 前缀：`【第一章 > 1.2 安装】\n正文`
- `source_location.section_title` 供引用卡片展示

### 避免切断关键语义

- 滑窗时在句末标点、逗号、空格处回退
- 编号列表优先在列表项之间切
- 过短块（< min_chunk_size）合并到相邻块

### 空行 / 特殊字符

- 连续空行压缩为 `\n\n`
- `\r\n` 统一为 `\n`
- 纯空白段落跳过

## 5. 与引用源展示的关系

引用卡片需回答：

1. **来自哪篇文档？** → `knowledge_item_id` + `title`
2. **什么位置？** → `page_start` / `section_title` / `heading_path`
3. **原文上下文？** → `content` + `char_start/end` 高亮

### source_location 示例

```json
{
  "page_start": 5,
  "page_end": 5,
  "section_index": 2,
  "section_title": "环境要求",
  "heading_path": ["安装指南", "环境要求"],
  "char_start": 1240,
  "char_end": 1580
}
```

### RAG Answer Reference

```python
AnswerReference(
    chunk_id=...,
    knowledge_item_id=...,
    item_title=...,
    excerpt=chunk.content[:200],
    source_location=...,
    score=...,
)
```

LLM 回答脚注 `[1][2]` 通过 `chunk_id` 关联，不让模型编造页码。

## 6. 实现建议

### Demo 阶段（当前）

- **结构化滑窗**：section → 段落 → 句子 → 字符滑窗
- 消费 `ParseResult.sections`；无 sections 时 fallback `raw_text`
- 手动录入条目：从兼容 `ParseResult` 后同样走切片

### 可后置增强

1. tiktoken 精确 token 计数
2. Parent-Child Chunk（小块检索、大块给 LLM）
3. 语义分段（embedding 相似度切分）
4. 增量 re-chunk + ChromaDB upsert
5. 按 file_type 调参

## 7. 代码结构与关联

```
backend/app/
  schemas/chunk.py           # SourceLocation, ChunkDraft, ChunkResponse
  models/chunk.py            # Chunk ORM
  services/chunk_service.py  # chunk_parse_result, apply_to_item
```

### 写入流程

```
ParserService.apply_to_item 成功
    → ChunkService.apply_to_item(db, item, parse_result)
        → 删除旧 chunks
        → chunk_parse_result(parse_result)
        → bulk insert chunks 表
        → item.chunk_count = len(chunks)
```

### 与向量库（下一阶段）

```python
vectors = embedding_service.embed([c.content for c in chunks])
chroma.upsert(
    ids=[c.chunk_id for c in chunks],
    embeddings=vectors,
    metadatas=[{"knowledge_item_id": ..., "order_index": ...}],
    documents=[c.content for c in chunks],
)
```

## 9. 面试表达

> 解析决定 RAG 上限，**chunk 决定检索粒度**。切太大向量语义稀释，切太小指代丢失。我们采用 section 边界 + 段落/句子合并 + 滑窗 fallback，中文默认 400 字、15% overlap，chunk 前缀注入标题路径。`source_location` 与 `content` 分开存，embedding 用 content，引用卡片用 page/section/char 偏移，回答脚注通过 chunk_id 关联。
