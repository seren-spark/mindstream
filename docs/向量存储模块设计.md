# 向量存储模块设计

> 对齐：`文本切片模块设计.md`、`knowledge-item-module.md`  
> 当前实现：`backend/app/services/vector_store_service.py` + `embedding_service.py`

---

## 1. 模块职责定义

### 双轨存储：各存什么

```
SQLite（关系库）                    ChromaDB（向量库）
─────────────────                  ─────────────────
knowledge_bases                    collection: kb_{id}
knowledge_items                    ├─ id = chunk.id (UUID)
chunks  ←── 1:1 映射 ──→           ├─ embedding (float[])
  · content 全文                     ├─ document (= chunk.content)
  · source_location JSON             └─ metadata (过滤 + 引用辅助)
  · chunk_metadata JSON
条目状态、标签、分类
```

| 存储 | 职责 |
|------|------|
| **SQLite** | 元数据 SSOT：条目、切片正文、引用定位、状态机、标签 |
| **ChromaDB** | 语义索引：embedding + 轻量 metadata，供 Top-K 检索 |

### 为什么要分开存

1. **职责分离**：关系库擅长 CRUD、事务、关联查询；向量库擅长近似最近邻检索
2. **引用完整性**：RAG 引用需 `source_location`、页码、章节等结构化 JSON，不适合全塞向量库
3. **独立演进**：可换 embedding 模型或向量引擎，而不动业务表结构
4. **过滤组合**：复杂标签/状态过滤先查 SQLite 得 `item_ids`，再向量库 `$in` 过滤

## 2. 向量存储结构设计

### Collection 设计

**每个知识库一个 Collection**，命名：`kb_{knowledge_base_id}`

| 方案 | Demo 选择 | 说明 |
|------|-----------|------|
| 全局单 collection | ❌ | 删除知识库需 where 扫描 |
| **每库一 collection** | ✅ | 隔离清晰，删库 = `delete_collection` |
| 每条目一 collection | ❌ | collection 数量爆炸 |

### 向量 ID 与 chunk.id 对齐

```
chunk.id (UUID) === ChromaDB document id
```

检索命中 `chunk_id` → 回查 SQLite `chunks` 表取完整 `source_location` → 渲染引用卡片。

### 写入字段

| 字段 | 来源 | 说明 |
|------|------|------|
| `ids` | `chunk.id` | 与 SQLite 主键一致 |
| `documents` | `chunk.content` | 冗余存储，检索时可快速预览 |
| `embeddings` | `EmbeddingService.embed_texts()` | 外部生成后写入 |
| `metadatas` | 见下表 | 仅 Chroma 支持的标量类型 |

### Metadata 字段（ChromaDB）

| 字段 | 类型 | 用途 |
|------|------|------|
| `knowledge_base_id` | int | 冗余校验 |
| `knowledge_item_id` | int | 按条目过滤 |
| `item_title` | str | 引用卡片标题（ denormalized ） |
| `order_index` | int | 同文档内排序 |
| `section_title` | str | 引用副标题 |
| `page_start` | int | PDF 页码（无则 0） |
| `item_status` | str | 检索默认过滤 `ready` |
| `source_type` | str | manual / file |
| `category` | str | 分类过滤 |
| `tags_csv` | str | 标签逗号拼接（Demo 简化） |

**不放入 ChromaDB 的字段：**

- `source_location` 完整 JSON（嵌套结构、列表）
- 长正文以外的业务字段（`summary`、`file_path`）
- 列表型 `tags`（Chroma 不支持 array metadata）

## 3. 元数据过滤设计

### 支持的过滤维度

| 维度 | 实现方式 | Demo |
|------|----------|------|
| `knowledge_base_id` | collection 隔离 | ✅ |
| `knowledge_item_id` | metadata `$eq` | ✅ |
| `item_status` | metadata `$eq`，默认 `ready` | ✅ |
| `category` | metadata `$eq` | ✅ |
| `source_type` | metadata `$eq` | ✅ |
| `tag` | 先 SQLite 查 item_ids，再 `$in` | ✅ |
| `knowledge_base.status` | 查 SQLite，disabled 库拒绝检索 | ✅ |

### 查询示例

```python
# 仅检索可用条目
where={"item_status": "ready"}

# 指定条目
where={"$and": [
    {"item_status": "ready"},
    {"knowledge_item_id": 42},
]}

# 指定标签（先查 SQLite）
item_ids = [1, 3, 5]  # tag='政策' 的条目
where={"$and": [
    {"item_status": "ready"},
    {"knowledge_item_id": {"$in": item_ids}},
]}
```

## 4. 数据一致性设计

### 写入时机

```
_finalize_ready → ChunkService.apply_to_item → VectorStoreService.sync_item_vectors
```

### 同步策略

| 事件 | SQLite | ChromaDB |
|------|--------|----------|
| 条目 process/ready | 重建 chunks | 删旧向量 + upsert 新向量 |
| 条目 disabled | status 更新 | `delete` where item_id |
| 删除条目 | CASCADE 删 chunks | `delete` where item_id |
| 删除知识库 | CASCADE | `delete_collection(kb_{id})` |
| 重建索引 | 不变 | 删 collection → 遍历 ready 条目 re-upsert |

### 一致性原则（generated
1. **SQLite 为 SSOT**：向量库是可重建的索引缓存
2. **先写 SQLite 再写向量**：chunk 落库 commit 后再 embed
3. **失败可重试**：提供 `POST .../vectors/rebuild` 全量重建
4. **条目级幂等**：sync 前先 `delete_by_item_id`，再 upsert

## 5. 封装设计

### VectorStoreService 方法

| 方法 | 职责 |
|------|------|
| `collection_name(kb_id)` | 生成 collection 名 |
| `upsert_chunks(item, drafts)` | 写入/更新条目向量 |
| `delete_item_vectors(kb_id, item_id)` | 删除条目全部向量 |
| `delete_knowledge_base_vectors(kb_id)` | 删除整个 collection |
| `query(kb_id, query_text, filter, top_k)` | 语义检索 |
| `get_stats(kb_id)` | 向量数量统计 |
| `rebuild_knowledge_base(db, kb_id)` | 全量重建索引 |
| `sync_item_from_db(db, item)` | 从 SQLite chunks 重建条目向量 |

### EmbeddingService

| 方法 | 职责 |
|------|------|
| `embed_texts(texts)` | 批量文本 → 向量 |
| `embed_query(text)` | 单条 query 向量 |

Demo 默认 `chromadb_default`（MiniLM）；测试可用 `hash` 模式（确定性伪向量，无语义）。

## 7. 为什么 Demo 选 ChromaDB

| 对比 | ChromaDB | Milvus / Qdrant / pgvector |
|------|----------|----------------------------|
| 部署 | 嵌入式 PersistentClient，零运维 | 需独立服务或扩展 |
| Python 集成 | 原生 API，与 FastAPI 同进程 | 客户端 + 运维成本 |
| Demo 数据量 | 千~万级 chunk 足够 | 大集群优势 Demo 用不上 |
| 面试表达 | 「索引层可替换，SSOT 在 SQLite」 | 方案过重 |

生产可换 Qdrant/pgvector，**接口边界在 VectorStoreService**。

## 8. 面试表达

> 我们采用双轨存储：SQLite 存 chunk 正文和引用定位，ChromaDB 只存 embedding 和检索用 metadata。向量 id 与 chunk.id 对齐，命中后回表取 source_location 做引用卡片。Demo 用嵌入式 ChromaDB 降低运维；一致性上 SQLite 是 SSOT，向量库可 rebuild。删条目先删向量再删库，或 rebuild API 兜底。

## 9. 验证清单

1. 条目 process 后 `vectors/stats` 的 count > 0
2. `vectors/search` 能返回 chunk_id + score + excerpt
3. 删除条目后向量 count 减少
4. 删除知识库后 collection 不存在
5. `vectors/rebuild` 后 count 与 SQLite chunks 一致
