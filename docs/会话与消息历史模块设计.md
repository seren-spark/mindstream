# 会话与消息历史模块设计

> 对齐：`RAG-Prompt组装模块设计.md`、`流式问答模块设计.md`、`引用源展示模块设计.md`  
> 现状：**已实现 P0+P1** — `conversations` / `conversation_messages` 表、REST API、流式落库、左侧会话列表、URL `?conversationId=`  
> 历史窗口：前端 `slice(-6)` + 后端 `trim_history`（`rag_max_history_turns=3`）

---

## 1. 模块职责

### 为什么知识问答也需要会话

| 问题 | 无会话 | 有会话 |
|------|--------|--------|
| 多轮指代 | 「它」「上一段」无法理解 | 同一 thread 内历史可解释 |
| 刷新页面 | 消息丢失 | 从 DB 恢复 |
| 审计与复盘 | 无法追溯 | 按会话查看问答记录 |
| RAG 上下文 | 每次单轮 | 滑动窗口带入最近 N 轮 |

### 会话 vs 消息

| 实体 | 解决什么 |
|------|----------|
| **Conversation（会话）** | 一次连续对话的容器：绑定知识库、标题、时间、消息条数 |
| **Message（消息）** | 单条 user/assistant 记录：内容、状态、引用、与流式/SSE 对齐 |

**不做复杂 IM**：无已读、无群聊、无在线状态；仅「知识库问答线程」。

## 2. 数据模型设计

### 2.1 `conversations` 表

```sql
CREATE TABLE conversations (
  id              VARCHAR(36) PRIMARY KEY,   -- UUID
  knowledge_base_id INTEGER NOT NULL REFERENCES knowledge_bases(id) ON DELETE CASCADE,
  title           VARCHAR(200) NOT NULL DEFAULT '新对话',
  message_count   INTEGER NOT NULL DEFAULT 0,
  last_message_at TIMESTAMP,
  created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  -- Demo 阶段单用户可省略 user_id；多租户再加
  -- user_id       VARCHAR(64),
  INDEX idx_conv_kb_updated (knowledge_base_id, updated_at DESC)
);
```

| 字段 | 说明 |
|------|------|
| `id` | 会话 UUID，前端路由 `/chat?c={id}` 或 path param |
| `knowledge_base_id` | 会话归属知识库，切换 KB 应新建或过滤列表 |
| `title` | 首条 user 问题截断 30 字，或可编辑 |
| `message_count` | 冗余计数，列表展示用 |
| `last_message_at` | 列表排序 |

### 2.2 `messages` 表

```sql
CREATE TABLE messages (
  id               VARCHAR(36) PRIMARY KEY,
  conversation_id  VARCHAR(36) NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  role             VARCHAR(20) NOT NULL,  -- user | assistant
  content          TEXT NOT NULL,
  status           VARCHAR(20) NOT NULL DEFAULT 'done',  -- pending|streaming|done|error|cancelled
  error_message    TEXT,
  references_json  JSON,                  -- CitationRef[]，仅 assistant 有值
  token_count      INTEGER,               -- 可选，后续成本统计
  created_at       TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at       TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_msg_conv_created (conversation_id, created_at ASC)
);
```

### 2.3 `references_json` 是否放在 message 中

**推荐：放在 `messages` 表，assistant 行，`references_json` 字段。**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 存在 message | 恢复历史时引用卡片一并展示；与 SSE `references` 事件一一对应 | JSON 略大 |
| 独立 citation 表 | 规范化 | Demo 过度设计 |
| 不存 | 刷新后无法展示引用 | 不符合 RAG 可解释性 |

序列化格式与前端 `CitationRef[]`、后端 `CitationRef` Pydantic 模型一致。

### 2.4 字段设计说明

| 字段 | 设计 |
|------|------|
| `role` | `user` / `assistant`（与 Prompt `ChatRole` 对齐，不含 system） |
| `status` | 流式中可写 `streaming`，完成后 `done`；失败 `error`；用户停止 `cancelled` |
| `created_at` | 列表排序、历史拉取；user 消息创建时即写入，assistant 可在 `start` 时占位 |
| 前端 `id` | 可与 DB `id` 统一用 UUID；`start` 事件返回 `message_id` 写 assistant 行 |


| 决策 | Demo 建议 |
|------|-----------|
| 左侧会话列表 | **要**，宽度 240px，按 `last_message_at` 降序 |
| 新建会话 | 顶栏或列表顶部「新对话」→ 创建 API → 清空消息区 |

### 刷新后保留历史

| 层级 | 策略 |
|------|------|
| **持久化（推荐）** | 消息落库；刷新后根据 URL `conversationId` 拉历史 |
| **localStorage（备选）** | Demo 零后端改造时缓存 `{ kbId, conversationId, messages }` |
| **仅内存（现状）** | 刷新丢失，不适合答辩 |

**推荐**：URL 带 `?conversationId=uuid`，刷新自动恢复。

## 4. 历史窗口策略

### 与现有实现对齐

| 位置 | 策略 |
|------|------|
| 前端 `useChatStream` | `slice(-6)` 条 done 消息（约 3 轮） |
| 后端 `trim_history` | `rag_max_history_turns=3`（最近 3 轮 user+assistant 对） |
| 检索 | 每轮仍用**当前 query** 做混合检索，历史只进 Prompt 不进向量 query（Demo 阶段） |

### 每轮带几轮历史合适

- **Demo / 答辩**：**3 轮**（6 条消息），与 `config.rag_max_history_turns` 一致  
- 原因：控制 token、指代够用、实现已有  

### 滑动窗口

```
DB 全量消息 ──加载──> 前端 messages[]
                         │
发送时 ──────────────────┼──> 取最近 N 轮 ──> POST history[]
                         │
后端 PromptBuilder.trim_history() 再次截断（双保险）
```

### 消息摘要压缩

| 阶段 | 方案 |
|------|------|
| **Demo** | **不做**；仅滑动窗口 |
| 二期 | 超过 10 轮时，旧轮用 LLM 摘要成一条 system 摘要消息 |

## 5. API 设计

### 路由前缀

`/api/knowledge-bases/{knowledge_base_id}/conversations`

### 接口列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/conversations` | 创建会话，返回 `{ id, title, knowledge_base_id }` |
| `GET` | `/conversations` | 列表，`?page=1&page_size=20`，按 `updated_at` 降序 |
| `GET` | `/conversations/{conversation_id}` | 会话详情 |
| `GET` | `/conversations/{conversation_id}/messages` | 消息历史，`?limit=100` |
| `POST` | `/conversations/{conversation_id}/messages` | 可选：非流式补录；流式场景见下 |
| `PATCH` | `/conversations/{conversation_id}` | 重命名标题 |
| `DELETE` | `/conversations/{conversation_id}` | 删除会话及消息 |

### 流式问答与存库（推荐时序）

```
1. POST /conversations（若无 conversation_id）
2. POST .../messages  { role: user, content }  → 落库 user 消息
3. POST .../chat/stream { query, history, conversation_id? }
   - start  → 创建 assistant 占位 (status=streaming)
   - token  → 仅内存/可选增量写 content（Demo 可结束后一次写入）
   - references → 更新 assistant.references_json
   - done   → status=done，更新 conversation.last_message_at
4. GET messages 用于切换会话恢复
```

`ChatStreamRequest` 扩展：

```python
class ChatStreamRequest(BaseModel):
    query: str
    history: list[ChatMessage] = []
    conversation_id: str | None = None  # 有则服务端可校验并持久化
    top_k: int = 5
```

### 6.1 SQLAlchemy 模型（示例）

```python
# backend/app/models/conversation.py
class Conversation(Base, TimestampMixin):
    __tablename__ = "conversations"
    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    knowledge_base_id: Mapped[int] = mapped_column(ForeignKey("knowledge_bases.id", ondelete="CASCADE"), index=True)
    title: Mapped[str] = mapped_column(String(200), default="新对话")
    message_count: Mapped[int] = mapped_column(Integer, default=0)
    last_message_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)

# backend/app/models/chat_message.py
class ChatMessageRecord(Base, TimestampMixin):
    __tablename__ = "messages"
    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    conversation_id: Mapped[str] = mapped_column(ForeignKey("conversations.id", ondelete="CASCADE"), index=True)
    role: Mapped[str] = mapped_column(String(20))
    content: Mapped[str] = mapped_column(Text)
    status: Mapped[str] = mapped_column(String(20), default="done")
    error_message: Mapped[str | None] = mapped_column(Text, nullable=True)
    references_json: Mapped[list | None] = mapped_column(JSON, nullable=True)
```

### 6.3 前端 Pinia Store（示例）

```typescript
// stores/chat-session.ts
export const useChatSessionStore = defineStore('chatSession', () => {
  const knowledgeBaseId = ref<number>()
  const conversationId = ref<string | null>(null)
  const conversations = ref<ConversationSummary[]>([])
  const messages = ref<ChatMessage[]>([])
  const loading = ref(false)

  async function ensureConversation(kbId: number) {
    if (conversationId.value) return conversationId.value
    const conv = await api.createConversation(kbId)
    conversationId.value = conv.id
    return conv.id
  }

  async function loadConversation(id: string) {
    loading.value = true
    try {
      const list = await api.getMessages(conversationIdToKb(id), id)
      messages.value = list.map(mapRecordToUiMessage)
      conversationId.value = id
    } finally {
      loading.value = false
    }
  }

  async function loadConversationList(kbId: number) {
    conversations.value = await api.listConversations(kbId)
  }

  function startNewConversation() {
    conversationId.value = null
    messages.value = []
  }

  function historyForRag(): ChatMessageHistory[] {
    return messages.value
      .filter((m) => m.status === 'done' && m.content)
      .slice(-6)
      .map((m) => ({ role: m.role, content: m.content }))
  }

  return {
    knowledgeBaseId,
    conversationId,
    conversations,
    messages,
    loading,
    ensureConversation,
    loadConversation,
    loadConversationList,
    startNewConversation,
    historyForRag,
  }
})
```

## 7. 面试表达

### 多轮理解

> 我们把「会话」和「消息」拆开：会话是会话线程，消息是 user/assistant 明细。多轮时前端从当前会话取最近几轮 done 消息作为 `history` 传给流式接口，后端 `trim_history` 再按轮次截断，和 RAG 检索用的当前 query 分离——检索始终针对最新问题，历史只进 Prompt 解决指代和连贯性。

### 历史窗口与成本

> Demo 阶段用**固定 3 轮滑动窗口**，前后端各截断一次，避免 token 爆炸。引用存在 assistant 消息的 JSON 字段里，刷新后仍能展示依据。没有做摘要压缩，答辩时可以说二期对长会话用摘要替代更早的轮次。会话落库后 URL 带 `conversationId`，刷新可恢复，体现产品完整度而不是单次内存数组。
